Diagnosing Login Failures Caused by Disk Exhaustion in a High-Turnover Domain Environment

A Windows Profile Service failure in a justice facility environment traced to accumulated stale user profiles consuming available disk space — and how it was resolved.

Article details

Category: Infrastructure / Endpoint Operations

Environment: Windows 11, domain-joined endpoints, high-turnover government facility

Tools used: WizTree, Windows Registry Editor, Computer Management

← Back to Writing

The symptom

Users at a correctional and criminal justice facility began receiving a "Windows Profile Service failed to start" error on login, preventing access to their workstations despite entering valid domain credentials. The affected machines were Windows 11 endpoints with 256GB of storage and 8GB of RAM, joined to the county domain.

The diagnosis

Initial investigation using WizTree — a disk space visualization tool — revealed that approximately 120GB of available storage was consumed entirely by accumulated local user profiles. The facility had high staff turnover, and domain profiles for former employees and temporary staff dating back to 2023 and early 2024 had been downloaded to the local machine and never cleaned up.

With the disk near capacity, Windows could not write the temporary data required to initialize a new profile session. The Profile Service fails silently in this condition — the error message points to the service itself rather than disk space, which makes the root cause non-obvious without a disk inspection step.

The resolution

Resolution involved three steps. First, identifying which profiles belonged to departed users by cross-referencing the profile folders in C:\Users against active accounts. Second, deleting the stale profile folders and removing the corresponding registry keys from:

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList

Each profile has a subkey identified by its Security Identifier (SID). Removing both the folder and the registry entry cleanly deregisters the profile from the system. After restart, the freed disk space allowed Windows to initialize profile sessions normally and affected users could log in.

Related case: corrupted user profile

A secondary related case involved a user whose profile had become corrupted rather than simply accumulated. In this scenario the ProfileList registry key contained a malformed or duplicate entry for the user's SID. Resolution required locating the correct ProfileGUID entry, correcting the path reference, and in some cases rebuilding the profile entirely by renaming the existing profile folder and allowing Windows to generate a fresh one on next login.

Operational lesson

Disk capacity is a silent dependency for Windows authentication that does not surface until it fails completely. In high-turnover environments where domain profiles accumulate on shared endpoints, periodic profile cleanup should be part of standard maintenance. Automated profile management through Group Policy — specifically the Delete cached copies of roaming profiles and Limit profile size settings — can prevent this class of failure from recurring without manual intervention.
